# Improved Prompt

You are generating a JSON evaluation dataset of exactly 150 fashion search queries for a fashion recommendation system. The dataset will be used to benchmark retrieval quality across three complexity tiers.

Return only one valid JSON array with 150 objects. Do not include markdown, comments, headings, or explanations.

Each object must use this exact schema:
```json
{
  "id": "q001",
  "query": "<natural language query>",
  "tier": 1,
  "tags": ["category", "color"]
}
```

Field rules:
- `id` must run sequentially from `q001` to `q150`.
- `tier` must be an integer: `1`, `2`, or `3`.
- `query` must be natural user language in English.
- `tags` must contain only the constraint types explicitly present in the query, chosen from:
  - `category`
  - `color`
  - `fit`
  - `price`
  - `occasion`

Tier structure:

Tier 1: Basic
- Count: 50 queries
- IDs: `q001` to `q050`
- Constraints per query: exactly 1 or 2
- No occasion
- Short, keyword-like or simple natural phrasing
- Examples of style only: `blue jeans`, `oversized black coat`, `I'm looking for a red dress`

Tier 2: Medium
- Count: 50 queries
- IDs: `q051` to `q100`
- Constraints per query: 3 or 4
- No occasion
- More natural phrasing than Tier 1
- Must include:
  - one category
  - one color
  - and at least one of: fit, price, second color

Tier 3: Complex
- Count: 50 queries
- IDs: `q101` to `q150`
- Must include one occasion plus at least 2 other constraints from: category, color, fit, price
- Most natural and conversational phrasing
- The occasion must be clearly inferable from the wording, but the query should still sound like a real user request

Use only these category values or close user phrasing that maps cleanly to them:
- jackets
- coats
- dresses
- tops
- shirts
- bodies
- jeans
- trousers
- t-shirts
- cardigans
- skirts
- sweatshirts
- shoes
- bags

Use colors grounded in the actual `offline/data/women_data.parquet` vocabulary. You may use exact values or realistic user-facing variants/synonyms that still map back to the catalog. Prioritize coverage across this dataset-derived palette:
- black, white, blue, red, green, pink, beige, brown, grey, navy blue, denim blue, burgundy, camel, cream, ecru, khaki, lilac, mustard, orange, silver, yellow, violet, wine, chocolate brown, bottle green, olive green, sky blue, light blue, dark navy, dark red, dark green
- You may also use natural phrasing for multi-tone colors when realistic, such as `black and white`, `blue and grey`, or `brown and ecru`
- Do not invent color families that are not reasonably grounded in the catalog

Allowed fit/style descriptors:
- oversized
- slim
- skinny
- loose
- wide leg
- straight leg
- cropped
- midi
- maxi
- mini
- short
- long sleeve
- short sleeve

Allowed occasion targets for Tier 3 only.
The wording in the query should imply one of these exact internal labels:
- wedding_guest
- party_night_out
- work_office
- formal_event
- casual_everyday
- date_night
- winter_outerwear
- smart_casual
- navidad_dinner
- nochevieja_nye
- halloween
- oktoberfest

Occasion-writing rules:
- Do not invent other occasions.
- Use natural user phrasing, not the label itself unless the label would sound natural.
- Make each Tier 3 query map clearly to one and only one of the allowed occasion targets.

Price rules:
- Use realistic budgets between `EUR20` and `EUR250`
- Phrase them naturally, for example:
  - `under EUR50`
  - `max EUR80`
  - `around EUR100`
  - `no more than EUR150`
  - `budget is EUR200`
- Use `EUR` instead of the euro symbol to keep JSON output clean and encoding-safe

Diversity requirements across all 150 queries:
- Cover all 14 categories at least once
- Cover all 12 occasions across Tier 3
- Cover at least 80% of the available catalog color space using exact colors or close user synonyms
- Mix phrasing styles:
  - keyword-like
  - plain natural requests
  - conversational requests
- Roughly one third of all queries should include a price constraint
- Roughly two thirds should not include a price constraint
- Do not repeat the same `category + color + fit` combination more than once
- Avoid near-duplicate wording templates across the dataset

Tagging rules:
- If the query contains a category, include `category` in `tags`
- If the query contains a color, include `color` in `tags`
- If the query contains a fit/style descriptor, include `fit` in `tags`
- If the query contains a budget or price limit, include `price` in `tags`
- If the query implies one of the allowed occasions, include `occasion` in `tags`
- `tags` must reflect what is actually present in the query text, not hidden metadata

Validation requirements before finalizing:
- There must be exactly 150 objects
- IDs must be sequential and unique
- Tier counts must be exactly 50 / 50 / 50
- Tier 1 and Tier 2 must contain no occasion
- Every Tier 3 query must include `occasion` in `tags`
- Every Tier 2 query must include `category` and `color` in `tags`
- Output must be valid JSON that can be parsed directly

Return only the JSON array.

## Alternative Approaches
1. Add per-tier mini-checklists inside the prompt so the model self-validates counts before emitting the JSON.
2. Require a hidden planning table first and then the final JSON array second, if you are using a model that handles multi-step structured generation reliably.
3. Split generation into three separate prompts by tier and merge the arrays afterward, if you want tighter control over balance and diversity.

## Validation Checklist
- Exactly 150 objects are returned
- IDs are unique and sequential from `q001` to `q150`
- Tier counts are 50 / 50 / 50
- Tier 1 contains no `occasion` tags
- Tier 2 contains no `occasion` tags
- Tier 3 contains `occasion` in every item's `tags`
- All 14 categories appear at least once
- All 12 allowed occasions appear at least once in Tier 3
- About one third of queries contain a price constraint
- No repeated `category + color + fit` combinations occur
- Output is valid JSON and contains no prose outside the array

## Edge Cases
- The model may overuse a few common categories like `dresses`, `jeans`, and `trousers` and undersample `bodies`, `bags`, or `shoes`.
- The model may drift into invented occasion labels such as `vacation`, `graduation`, or `brunch`.
- Tier 2 queries may accidentally become too simple if they omit a true third constraint.
- Tier 3 queries may include vague social settings that do not map cleanly to one allowed internal occasion.
- Color coverage may collapse into a small set of popular tones unless explicitly watched.

## What Could Go Wrong
- The model may output invalid JSON if it mixes explanations with the array.
- The model may satisfy the tier counts but fail the diversity constraints.
- The tags may describe inferred metadata instead of only what appears in the query text.
- The euro symbol may create inconsistent encoding or escaping in some model outputs.
